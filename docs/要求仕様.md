# 要求仕様書

## 1. 目的・背景

スマートフォン内蔵カメラが高画質化する一方で、写真表現の画一化が進んでいる。本プロダクトは自作カメラとそれを操作する専用アプリを組み合わせ、「レトロな質感の写真」を撮影・編集・SNS共有するまでを一気通貫で体験できる環境を提供する。

- **解決したい課題**: スマホカメラでは得られないレトロ感ある写真を、手軽に撮影してSNSへ流したい
- **提供価値**: 自作カメラの撮影体験 × ワンストップのSNS共有フロー
- **ゴール**: 自作カメラ・スマホアプリ・サーバの三者が連携する最小構成のプロトタイプを完成させる

## 2. 対象ユーザー

- **年齢層**: SNSへ気軽に写真を投稿し、レトロな雰囲気を好む若年層
- **利用場所**: 屋外、明るい場所
- **利用シーン**: 写真を撮りたいとき、撮影した写真をすぐSNSへ投稿したいとき

## 3. 利用環境

- **自作カメラ**: 撮影本体。GPSは搭載しない
- **サーバ**: カメラ・スマホ・PCを仲介する中継点。画像保管・サムネ生成・配信を担う
- **スマートフォン**: Android / iPhone（操作・閲覧・SNS投稿の主端末）
- **PC**: Mac / Windows（補助的に画像閲覧・転送で利用想定）
- **接続方式**:
  - カメラ ⇔ スマホ: **BLE**（シャッター操作・カメラ状態取得など軽量な制御信号）
  - カメラ ⇔ サーバ: **Wi-Fi**（撮影画像のアップロード）
  - サーバ ⇔ スマホ / PC: **Wi-Fi または モバイル回線**（画像ダウンロード等）
  - 補助: **SDカードの物理取り出し**による読み出しも許容

## 4. 画面仕様

GUIで提供する主な画面・要素を以下に挙げる。

- 写真の閲覧画面（撮影日時順グリッド／位置情報マップ表示）
- 転送画面（サーバ → スマホ／PC）
- 簡単なレタッチ画面
- SNSへの直接投稿画面
- 異次元の圧縮（コンセプト段階・要調査）
- 撮影場所のGPS表示
- AIによる画像選定（コンセプト段階・要調査）

## 5. 機能要件

各機能の概要を以下に示す。詳細データ仕様は別紙にて補足する想定。

### 5.1 位置情報取得機能

- スマホ ⇔ カメラ が BLE 接続中、シャッター押下時にスマホのGPS情報を取得する
- 取得した位置情報は画像IDと紐付けてアプリのSQLiteに保存（画像ファイルのEXIFには埋め込まない）
- カメラ本体にはGPSを搭載しない
- SNS投稿時はそのまま位置情報を付けて投稿する

### 5.2 ログイン機能

- X（旧Twitter）の OAuth によるログインを主とし、SNS投稿の認可も兼ねる
- 未ログイン状態でも利用できるゲストモードを提供する（撮影・閲覧・レタッチは可、SNS投稿は不可）
- アクセストークン／リフレッシュトークンはアプリのSQLiteに保存する
- 期限切れ時はリフレッシュトークンを用いて自動更新する

### 5.3 画像表示機能

- 表示元
  - スマホに転送済みの画像（ローカル）
  - PCに転送済みの画像
  - カメラ本体は自身の画面上でのみ表示可能
- 一覧表示
  - 撮影日時順のグリッド表示
  - 位置情報を持つ画像は地図上にピンとしても表示可能（位置情報のない画像は非表示）
- サムネイル
  - サーバ側で生成し、サーバからアプリへ配信する

### 5.4 Codex app server 連携機能

- 構成: **カメラ ⇔ サーバ ⇔ スマホアプリ** の3層構成。カメラ自身はサーバではない
- 通信プロトコル: **HTTP / REST API**
- 認証: 同一アカウントによる認証。カメラ・スマホアプリの双方がサーバへ同一アカウントでログインし、それを根拠に紐付け／接続する

### 5.5 カメラ連携機能（BLE）

- 初回はOS標準のBLEペアリングを行い、以降は自動で再接続する
- スマホ ⇔ カメラ間でやり取りする制御コマンド:
  - シャッター押下
  - バッテリー残量・ストレージ残量の取得
- 接続切断時のリトライ等の挙動については本要件では規定しない

### 5.6 撮影機能

- 撮影画像の出力ファイル形式: **JPEG + RAW** の両方
- レトロ調の画像加工はカメラ・サーバ側では行わず、スマホアプリの閲覧／レタッチ時に適用する
- 撮影完了後、スマホアプリに対しサーバ経由でサムネを即時表示する

### 5.7 SDカード保存機能

- 撮影画像は **常にSDカードとサーバの両方** に保存する
- SD内のフォルダ・ファイル命名は **DCIM準拠**（一般的なカメラと同じ構造）
- 容量満杯時の挙動は本要件では規定しない

### 5.8 画像転送機能

- サーバ上にある画像を、ユーザー操作によりスマホまたはPCへダウンロードする
- スマホ・PCのいずれにおいても、ユーザーが一覧から対象を選択してダウンロードする方式とする
- 自動同期は行わない

### 5.9 レタッチ機能

- スマホアプリ上で画像の編集を行う
- 提供する加工
  - レトロ調プリセットフィルタ（複数種類）
  - 明るさ・コントラスト等の微調整
- 編集結果は元画像とは別ファイルとして保存し、元画像を破壊しない

### 5.10 SNS投稿機能

- X（旧Twitter）への直接投稿を行う（投稿認可は 5.2 のOAuthを流用）
- 投稿フロー: **キャプション入力 → 位置情報を含めるか選択 → 投稿**
- 位置情報を含めるかは投稿ごとに選択可能

## 6. 非機能要件

保留

## 7. データ要件

- SQLite（スマホアプリ側のローカルデータ保持に使用）

## 8. 外部連携要件

- **X（旧Twitter）への投稿連携**: OAuth経由で認可を取得し、アプリから直接投稿を行う
- そのほか任意のSNSアプリと連携できると望ましい（実装可否は要調査）
