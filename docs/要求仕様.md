# 要求仕様書

## 1. 目的・背景

### 1.1 撮影体験としての動機

スマートフォンのカメラは便利だが、「撮る」という行為そのものには最適化されておらず、シャッターを切る感覚や所有の満足感が薄い。一方、既存のトイカメラはネットワークに繋がらず取り回しが煩雑、デジカメは高くて手が出ない。
本プロダクトは「**撮ることそれ自体を楽しめる専用機**としての満足感」と「撮ったらすぐ手元の他のデバイスで扱える現代的な利便性」を両立させたカメラを目指す。

### 1.2 自作プロジェクトとしての動機

本プロジェクトは「**HW から SW まで一貫して自分たちの手で作ること**」自体を目的のひとつとする。
普段の業務では分業されがちな、回路・筐体・組み込みファームウェア・OS・アプリケーション・連携先（スマホアプリやクラウド）までを、同じチームで地続きに設計・実装していく経験を得たい。物理と論理の境目で初めて顕在化する問題（電源管理、シャッターのレイテンシ、放熱、操作感、ノイズ等）に直接向き合えるのは、既製品ベースの開発では得難い学びであり、面白さでもある。また、既製品の制約に縛られずに「自分たちが欲しい撮影体験」をそのまま形にできることも、自作する動機のひとつである。

### 1.3 体験を後押しする副次価値

撮影体験を中心軸に据えつつ、撮影後の取り回しとして以下の副次価値を提供する。

- **レトロな質感の表現**: 専用機ならではの色作りをレタッチで提供
- **ワンストップの SNS 共有**: 撮影 → スマホ転送 → 投稿までの導線を短くする

### 1.4 ゴール

自作カメラ（メイン）・自作操作機（サブ）・スマホアプリ・サーバの **4 要素が連携する最小構成のプロトタイプを完成させる**。

## 2. 対象ユーザー

- **年齢層**: SNSへ気軽に写真を投稿し、レトロな雰囲気を好む若年層
- **利用場所**: 屋外、明るい場所
- **利用シーン**: 写真を撮りたいとき、撮影した写真をすぐSNSへ投稿したいとき

## 3. 利用環境

- **自作カメラ（メインモジュール）**: 撮影本体。EVF、タッチパネル、物理ボタン、64MPカメラモジュールを備える。GPSは搭載しない
- **自作操作機（サブモジュール）**: 本体から離れた位置でシャッターと十字キーを操作するための小型リモコン
- **サーバ**: カメラ・スマホ・PCを仲介する中継点。画像保管・サムネ生成・配信を担う
- **スマートフォン**: Android / iPhone（閲覧・SNS投稿の主端末。BLEで位置タグ付けにも参加）
- **PC**: Mac / Windows（補助的に画像閲覧・転送で利用想定）
- **接続方式**:
  - カメラ ⇔ サブモジュール: **BLE**（シャッター操作・十字キー）
  - カメラ ⇔ スマホ: **BLE**（撮影完了通知の受信と位置タグ付け、スマホからのリモートシャッター）
  - カメラ ⇔ サーバ: **Wi-Fi**（撮影画像のアップロード）
  - サーバ ⇔ スマホ / PC: **Wi-Fi または モバイル回線**（画像ダウンロード等）
  - 補助: **SDカードの物理取り出し**による読み出しも許容

## 4. 画面仕様

### 4.1 カメラ本体（メインモジュール）

- **EVF（電子ビューファインダー）**: ファインダーを覗いて構図決定する経路
- **タッチパネル**: 背面に配置。構図確認・撮影画像の再生・設定UI
- **物理ボタン**: 電源、シャッター（2段押し）、十字キー、メニュー、再生（主たる操作は物理ボタンで完結）
- **撮影モード画面**: ライブビュー（EVFまたはタッチパネル）
- **再生モード画面**: 撮影済み画像の一覧／詳細
- **設定画面**: 撮影モード、ISO、SS、WB、ファイル形式 等
- **セットアップ画面**: 初回起動時の Wi-Fi 設定、X ログイン誘導

### 4.2 スマホ／PC アプリ（Tauri）

- 写真の閲覧画面（撮影日時順グリッド／位置情報マップ表示）
- 転送画面（サーバ → スマホ／PC）
- 簡単なレタッチ画面
- SNSへの直接投稿画面
- 異次元の圧縮（コンセプト段階・要調査）
- 撮影場所のGPS表示
- AIによる画像選定（コンセプト段階・要調査）

### 4.3 サブモジュール（操作機）

GUIは持たず、物理操作のみを提供する。

- 2段シャッターボタン（半押し／全押し）
- 十字キー（4方向 + 中央押し込み）
- 電源スイッチ
- 接続状態を示すLED（任意）

## 5. 機能要件

各機能の概要を以下に示す。詳細データ仕様は別紙にて補足する想定。

### 5.1 電源機能

- メインモジュールの電源 ON / OFF を物理スイッチで行う
- 起動時に通常モード／セットアップモードを判定し、対応する画面を表示
- 一定時間の無操作で自動スリープ（任意機能）

### 5.2 構図決定機能

- ライブビューを EVF とタッチパネルの **両方に常時表示** する（切替機構は持たない）
- ユーザーは物理ボタンまたはタッチ操作で構図を決定する
- 両方の表示が常時点灯することで消費電力は増えるが、操作の単純さとレスポンスを優先する

### 5.3 オートフォーカス（AF）機能

- シャッター半押しで AF を実行
- 64MPカメラモジュール（Arducam）のプログラマブルフォーカスを利用
- AF領域はライブビュー上で指定可能（タッチ AF）
- AF合焦時はビープ音またはインジケータで通知

### 5.4 撮影設定変更機能

- 撮影モード（オート／マニュアル）、ISO、シャッタースピード、ホワイトバランス、ファイル形式（JPEG / RAW / 両方）を変更できる
- 物理ボタンまたはタッチパネルで操作する
- 設定は不揮発に保持し、電源OFFしても維持される

### 5.5 位置情報取得機能

- カメラの **Capture Done** Notify をスマホアプリが BLE で受信したタイミングで、スマホのGPS情報を取得する
- 取得した位置情報は `capture_id` と紐付けてアプリのSQLiteに保存（画像ファイルのEXIFには埋め込まない）
- カメラ本体にはGPSを搭載しない
- スマホが BLE 接続していない瞬間に撮影された画像には位置情報が紐付かない
- SNS投稿時はそのまま位置情報を付けて投稿する

### 5.6 ログイン機能

- X（旧Twitter）の OAuth によるログインを主とし、SNS投稿の認可も兼ねる
- 未ログイン状態でも利用できるゲストモードを提供する（撮影・閲覧・レタッチは可、SNS投稿は不可）
- アクセストークン／リフレッシュトークンはアプリのSQLiteに保存する
- 期限切れ時はリフレッシュトークンを用いて自動更新する

### 5.7 画像表示機能

- 表示元
  - スマホに転送済みの画像（ローカル）
  - PCに転送済みの画像
  - カメラ本体は自身の画面上でのみ表示可能
- 一覧表示
  - 撮影日時順のグリッド表示
  - 位置情報を持つ画像は地図上にピンとしても表示可能（位置情報のない画像は非表示）
- サムネイル
  - **オリジナル画像のサムネ**: 撮影画像のアップロードを受けたサーバ側で生成し、MinIO に保存
  - **レタッチ画像のサムネ**: アプリ側（Tauri Core / Rust）で生成し、画像本体と一緒にサーバへアップロード
  - いずれもアプリは GET でサーバから取得する

### 5.8 Codex app server 連携機能

- 構成: **カメラ ⇔ サーバ ⇔ スマホアプリ** の3層構成。カメラ自身はサーバではない
- 通信プロトコル: **HTTP / REST API**
- 認証: 同一アカウントによる認証。カメラ・スマホアプリの双方がサーバへ同一アカウントでログインし、それを根拠に紐付け／接続する

### 5.9 BLE 連携機能

- カメラは BLE Peripheral として動作し、**サブモジュール**と**スマホアプリ**の両方から同時接続を受け付ける
- 初回はOS標準のBLEペアリングを行い、以降は自動で再接続する
- カメラから接続Central（サブモジュール／スマホ）へ送る通知:
  - 撮影完了（capture_id 付き）
  - バッテリー残量・ストレージ残量
- Centralからカメラへ送る制御:
  - シャッター半押し／全押し
  - 十字キー操作（サブモジュールのみ）
  - サブモジュール自身のバッテリー残量通知
- 接続切断時のリトライ等の挙動については本要件では規定しない

### 5.10 撮影機能

- 撮影のトリガー経路: メイン本体の物理シャッター／サブモジュールの物理シャッター（BLE）／スマホアプリのシャッター（BLE）
- 半押しでAF実行、全押しで撮影
- 撮影瞬間にカメラが `capture_id`（UUID v4）を生成し、画像メタとして付与
- 撮影画像の出力ファイル形式: **JPEG + RAW（DNG）** の両方
- レトロ調の画像加工はカメラ・サーバ側では行わず、スマホアプリの閲覧／レタッチ時に適用する
- 撮影完了後、BLE 経由で Capture Done を全接続Centralへ通知
- 画像本体は Wi-Fi 経由でサーバへアップロードする

### 5.11 SDカード保存機能

- 撮影画像は **常にSDカードとサーバの両方** に保存する
- SD内のフォルダ・ファイル命名は **DCIM準拠**（一般的なカメラと同じ構造）
- 容量満杯時の挙動は本要件では規定しない

### 5.12 画像転送機能

- サーバ上にある画像を、ユーザー操作によりスマホまたはPCへダウンロードする
- スマホ・PCのいずれにおいても、ユーザーが一覧から対象を選択してダウンロードする方式とする
- 自動同期は行わない

### 5.13 レタッチ機能

- スマホアプリ上で画像の編集を行う
- 提供する加工
  - レトロ調プリセットフィルタ（複数種類）
  - 明るさ・コントラスト等の微調整
- 編集結果は元画像とは別ファイルとして保存し、元画像を破壊しない

### 5.14 SNS投稿機能

- X（旧Twitter）への直接投稿を行う（投稿認可は 5.6 のOAuthを流用）
- 投稿フロー: **キャプション入力 → 位置情報を含めるか選択 → 投稿**
- 位置情報を含めるかは投稿ごとに選択可能

## 6. 非機能要件

撮影体験を損なわないために以下の目標値を満たす。具体値は試作・計測を経て調整する。

### 6.1 応答性

- 電源 ON から撮影可能までの起動時間: **10秒以内**
- シャッター全押しから撮影完了（記録）までのレイテンシ: **300ms以下**
- BLE Shutter Trigger 受信から撮影完了までのレイテンシ: **400ms以下**
- 撮影完了からスマホへの Capture Done Notify 到達: **200ms以下**

### 6.2 携帯性

- メインモジュール本体重量: **700g以下**（バッテリー込み）
- メインモジュール寸法: 一般的なAPS-Cミラーレス相当（W150 × H100 × D80 mm以内を目安）
- サブモジュール重量: **100g以下**

### 6.3 稼働時間

- メインモジュール: 1充電で **撮影500枚以上 または 連続使用4時間以上**
- サブモジュール: 1充電で **連続使用8時間以上**
- 待機（BLE接続維持のみ）: **数日**

### 6.4 接続性

- Wi-Fi: 802.11 a/b/g/n/ac 対応、2.4GHz / 5GHz 両対応
- BLE: 4.2 以上、複数Central同時接続
- モバイル回線経由のテザリングにも対応する

### 6.5 信頼性

- 撮影中のクラッシュ時もSDカード上のデータは失われない（書き込み完了まで電源断耐性を持つ）
- サーバ転送失敗時はカメラ内に保留し、Wi-Fi復帰時に再送する

## 7. データ要件

- **PostgreSQL**: サーバ側のメタデータ（ユーザー、カメラ、画像エントリ等）を保持
- **MinIO（S3互換オブジェクトストレージ）**: サーバ側の画像本体（JPEG / RAW / サムネ）を保持
- **SQLite**: スマホアプリ側のローカルデータ保持に使用（画像IDと位置情報の紐付け、Xトークン等）

## 8. 外部連携要件

- **X（旧Twitter）への投稿連携**: OAuth経由で認可を取得し、アプリから直接投稿を行う
- そのほか任意のSNSアプリと連携できると望ましい（実装可否は要調査）

## 9. セキュリティ要件

個人・学習目的プロトタイプとして、過剰な要件は課さないが、トークンと画像の取り扱いに最低限の配慮を行う。

### 9.1 認証・認可

- X の OAuth は PKCE 付きで実装し、`client_secret` を端末に持たない
- サーバセッショントークンはサーバ DB に **ハッシュ（SHA-256）** で保管し、生トークンは保持しない（[10.9.1 sessions](#1091-サーバpostgresql) 参照）
- セッショントークンは失効日時を持ち、サーバ側で強制失効が可能

### 9.2 トークン保管

- アプリ側の X アクセス／リフレッシュトークンは SQLite に保存。OS のセキュアストレージ移行は将来課題
- カメラ側の X リフレッシュトークンは `config.toml` 内に保存し、ファイル権限を `0600` に制限する

### 9.3 通信

- サーバとの通信は **HTTPS** を前提とする（家サーバ運用時も自己署名証明書 + ローカル CA で TLS 化）
- BLE 通信は暗号化なしのバイナリ通信。秘匿情報（トークン等）は BLE に流さない

### 9.4 端末・データ保護

- カメラ／サブモジュールの物理的盗難については本要件では考慮しない
- SD カードのデータは平文。紛失リスクはユーザーが受容するものとする

### 9.5 権限要求

- スマホアプリは Bluetooth・位置情報・通知の各権限を、利用時に都度ユーザーへ要求する
- 位置情報の利用は SNS 投稿時のオプトイン／オプトアウトを保証する

## 10. 技術仕様

本プロダクトを実装するための技術的な構成と各コンポーネントの仕様を示す。位置づけは個人・学習目的のプロトタイプであり、スケール／高可用性は要件外とする。

### 10.1 システム構成

```
 ┌────────────────┐                  ┌──────────────┐
 │  サブモジュール  │◀───── BLE ──────▶│              │
 │  (ESP32 / Rust)│  Shutter/Nav     │              │
 └────────────────┘                  │              │
                                     │              │       ┌────────────────┐
 ┌────────────────┐                  │   カメラ      │  Wi-Fi│                │
 │   スマホアプリ   │◀───── BLE ──────▶│ (Pi5 / Rust) │──────▶│   サーバ        │
 │    (Tauri)     │  CaptureDone等   │              │       │ (Go + Gin)     │
 └───────┬────────┘                  └──────────────┘       │                │
         │                                                  │  ┌──────────┐  │
         │            Wi-Fi (REST/JSON)                     │  │PostgreSQL│  │
         └─────────────────────────────────────────────────▶│  └──────────┘  │
                                                            │  ┌──────────┐  │
 ┌────────────────┐                                         │  │  MinIO   │  │
 │    PCアプリ     │◀──────── Wi-Fi (REST/JSON) ────────────▶│  └──────────┘  │
 │    (Tauri)     │                                         └────────────────┘
 └────────────────┘
```

- BLE 経路は2系統。**カメラ ⇔ サブモジュール**（シャッター/十字キー）と **カメラ ⇔ スマホ**（CaptureDone 受信・リモートシャッター）。カメラは Peripheral で複数 Central 同時接続を許容
- Wi-Fi（HTTP/REST）は **サーバを中心としたハブアンドスポーク**。画像本体やメタデータの大容量通信はすべてサーバ経由
- カメラ・スマホアプリは同一アカウントでサーバへログインすることで紐付く
- スマホアプリと PC アプリは同一の Tauri 実装をマルチプラットフォームでビルドする
- サブモジュールはネットワークを持たず BLE のみで動作する

### 10.3 サーバ仕様

#### 役割

- カメラから撮影画像（JPEG / RAW）を受け取り、MinIO へ保存
- 受け取った画像のサムネを生成（リサイズ）し MinIO へ保存
- 画像メタデータ（ユーザー／カメラ／撮影日時等）を PostgreSQL に保存
- スマホ／PCアプリへ画像一覧・本体・サムネを配信
- ユーザー／カメラの紐付け管理

#### 設計方針

- Gin によるシンプルな REST API
- GORM によるモデル定義とマイグレーション
- MinIO への接続は AWS SDK for Go（v2）を MinIO エンドポイントで利用
- 画像は **ユーザーID/カメラID/年月日/ファイルID** のキー設計でオブジェクト保存

### 10.4 カメラ仕様（メインモジュール）

#### 役割

- 物理シャッターおよび BLE 経由のリモートシャッター（サブモジュール／スマホ）で撮影
- 撮影画像を **SDカード（DCIM準拠）** と **サーバ（Wi-Fi経由）** の両方に保存
- BLE 経由で接続中の全Centralへ Capture Done を Notify
- バッテリー残量・ストレージ残量を BLE 経由で接続Centralに通知
- 自身のディスプレイで撮影済み画像を閲覧

#### ハードウェア構成

| 項目 | 採用品／仕様 |
|---|---|
| SoC | Raspberry Pi 5（または Pi 4） |
| カメラセンサ | Arducam 64MP AF カメラモジュール（プログラマブルフォーカス対応） |
| EVF | 小型EVFモジュール（Aliexpress 等で入手可能な汎用品） |
| タッチパネル | Pi対応の小型タッチディスプレイ（DSI または HDMI 接続） |
| ストレージ | microSD カード（DCIM 準拠で書き込み） |
| 無線 | Pi 内蔵 Wi-Fi（2.4/5GHz）および BLE |
| バッテリー | 自作モジュール（リチウムイオン + 昇圧 + 充電IC、USB-C で充電想定） |
| 物理ボタン | 電源、シャッター（2段）、十字キー、メニュー、再生切替 |

#### ソフトウェア構成

- 言語: **Rust**
- 撮影: `libcamera` の Rust バインディング（`libcamera-rs` 等）を利用。Rust バインディングが要件を満たさない場合は C++ FFI による自前バインディングも検討
- BLE Peripheral 実装: `bluer`（BlueZ ベース）
- 画像のレトロ加工はカメラ側では行わず生の JPEG/DNG をそのまま保存・送信
- ディスプレイ描画: framebuffer 直書き、または `embedded-graphics` 等
- ファームウェアは `systemd` サービスとして常駐

### 10.5 アプリ（Tauri）仕様

#### 役割

- BLE でカメラと接続し、必要に応じてリモートシャッターを送る
- BLE で受信した **Capture Done** Notify をトリガに、現在のGPSを取得し `capture_id` と紐付けて SQLite に保存
- サーバ（Wi-Fi）から画像一覧・サムネ・本体を取得して表示
- レタッチ機能（プリセットフィルタ／明るさコントラスト調整）
- X への OAuth ログインと SNS 投稿

#### 設計方針

- Tauri 2.x（Mac / Windows / iOS / Android のマルチターゲット）
- フロントエンド: React + TypeScript + Vite
- ネイティブ機能はコミュニティプラグインを利用（不足分は自前 Rust プラグインで補う）
- 地図表示: MapLibre GL JS（位置情報のある画像のみピン表示）
- 画像処理: Tauri command を介して **Rust 側で処理を実施**（`image` crate 等）。フロント側ではプレビュー用に CSS filter / Canvas で簡易表示
- ローカルDB: SQLite（`tauri-plugin-sql` 等を利用）

### 10.6 認証フロー

#### X OAuth（PKCE）

1. アプリが PKCE の `code_verifier` / `code_challenge` を生成
2. アプリがシステムブラウザ（または WebView）で X の認可エンドポイントを開く
3. ユーザーが認可すると、登録済みリダイレクトURI（カスタムスキーム）でアプリに `code` が返る
4. アプリが `code` + `code_verifier` を X のトークンエンドポイントに送り、access/refresh token を取得
5. トークンはアプリのローカル SQLite に保存
6. アクセストークン期限切れ時はリフレッシュトークンで自動更新

#### サーバへのログイン（カメラ／アプリ共通）

- 同一アカウントによる紐付け方式（具体実装は別途検討）
- アプリは X のユーザー識別をもとにサーバアカウントを作成／参照
- カメラも同一サーバアカウントへログイン（初期設定時にユーザーがカメラ画面から認証）

### 10.7 BLE プロトコル

#### 10.7.1 基本方針

- BLE は **シャッター／十字キー操作** と **状態通知**、そして **capture_id の配布** に用いる
- カメラが **Peripheral**、サブモジュールとスマホが **Central**。カメラは複数Centralの同時接続を許容
- ペイロードは **軽量バイナリ固定長**（BLE のデフォルト MTU 23 バイトに収まる範囲）
- 画像本体は BLE では一切流さない（Wi-Fi 経由でサーバへ）
- UUID は実装時に確定（v4 で固定値を一度生成し、すべての実装で共有）

#### 10.7.2 capture_id 設計

撮影ごとに一意のIDを発行し、**BLE / Wi-Fi / SQLite を貫通する識別子**とする。

- 形式: **UUID v4（16バイト）**
- 発行: **カメラ本体**が撮影瞬間に生成する（サブモジュール／スマホのどちらから引かれても、カメラが採番する一意ID）
- 流れ:
  1. Central（サブモジュール or スマホ）が Shutter Full を Write
  2. カメラが撮影と同時に capture_id を発番、画像メタとして保持
  3. カメラが Capture Done を全接続Centralへ Notify（status + capture_id）
  4. スマホは Capture Done を受信した瞬間に GPS を取得し、capture_id をキーに SQLite へ保存
  5. カメラはサーバへ画像アップロード時に capture_id を同梱
  6. スマホは後でサーバから画像一覧を取得し、capture_id でローカル GPS と突合
- スマホが BLE 接続していない瞬間に撮影された画像には GPS が付かない

#### 10.7.3 サービス／キャラクタリスティック定義

サービス: **Camera Control Service**

| キャラクタリスティック | 方向 | 種別 | ペイロード | 用途 |
|---|---|---|---|---|
| Shutter Half | Central → カメラ | Write | 1 バイト（0x00=release / 0x01=half-press） | AF始動／解除 |
| Shutter Full | Central → カメラ | Write | 1 バイト（0x01=trigger） | シャッター撮影指示 |
| Navigation Key | Central → カメラ | Write | 1 バイト（key code） | 十字キー操作（サブモジュール想定） |
| Capture Done | カメラ → Central | Notify | 17 バイト（status: 1 バイト + capture_id: 16 バイト） | 撮影完了通知。status は 0x00=成功 / 0x01=失敗 |
| Battery Status | カメラ → Central | Read / Notify | 1 バイト（0〜100 の整数: %） | カメラ本体のバッテリー残量 |
| Storage Status | カメラ → Central | Read / Notify | 1 バイト（0〜100 の整数: %） | SD残容量 |
| Submodule Battery | Central(サブ) → カメラ | Write | 1 バイト（0〜100 の整数: %） | サブモジュール自身のバッテリー残量を通知 |

Navigation Key の key code 例: `0x01=Up`, `0x02=Down`, `0x03=Left`, `0x04=Right`, `0x05=Center`, `0x06=Menu`, `0x07=Playback`

#### 10.7.4 通知タイミング

- **Battery Status / Storage Status**: 接続中は **10秒ごとに Notify**。値変化の有無にかかわらず送信
- **Capture Done**: 撮影完了時に1回 Notify。失敗時も status=0x01 で必ず通知
- **Submodule Battery**: サブモジュールが **60秒ごとに Write**

#### 10.7.5 ペアリング

- 初回ペアリングは Central の種別ごとに別フローを用いる
  - **スマホ**: OS 標準のペアリング UI
  - **サブモジュール**: カメラの設定画面からの承認フロー（詳細は [10.14.4](#10144-メインカメラとのペアリング) を参照）
- 以降はカメラ起動時に自動再接続
- 切断時のリトライ等は本要件では規定しない

### 10.8 API 設計

#### 10.8.1 共通仕様

- ベースURL: `/v1`
- 形式: JSON（Content-Type: `application/json`、アップロードのみ `multipart/form-data`）
- 認可: `Authorization: Bearer <session_token>` ヘッダ
- 時刻: すべて ISO 8601 / UTC（例: `2026-05-13T14:30:00Z`）
- ID: UUID v4 文字列
- ページネーション: カーソルベース（`before` / `after` クエリパラメータ + `limit`）

#### 10.8.2 エラー形式（RFC 7807 Problem Details 互換）

```json
{
  "type": "https://nukon.example.com/problems/not-found",
  "title": "Resource not found",
  "status": 404,
  "detail": "Image with id 5c8b... does not exist",
  "instance": "/v1/images/5c8b..."
}
```

`Content-Type: application/problem+json` で返す。

#### 10.8.3 エンドポイント一覧

| メソッド | パス | 概要 | 利用元 |
|---|---|---|---|
| POST | `/v1/auth/sessions` | X アクセストークンと引き換えにサーバセッションを発行 | アプリ／カメラ |
| POST | `/v1/cameras` | カメラを自分のアカウントへ登録 | カメラ |
| POST | `/v1/images` | 撮影画像のアップロード | カメラ |
| GET | `/v1/images` | 画像一覧（撮影日時降順、カーソルページング） | アプリ |
| GET | `/v1/images/by-capture-id/:capture_id` | capture_id から画像を取得 | アプリ |
| GET | `/v1/images/:id` | 画像メタデータ | アプリ |
| GET | `/v1/images/:id/file` | 画像本体／サムネのダウンロード | アプリ |
| DELETE | `/v1/images/:id` | 画像削除 | アプリ |

#### 10.8.4 各エンドポイントの詳細

##### POST /v1/auth/sessions

X の OAuth で得たアクセストークンをサーバへ提示し、サーバセッションを取得する。サーバは X のユーザー情報APIでトークンを検証し、対応するユーザーを作成または取得する。

- Request

```json
{
  "x_access_token": "AAAA..."
}
```

- Response 200

```json
{
  "session_token": "eyJ...",
  "user_id": "9f3c...",
  "expires_at": "2026-05-20T14:30:00Z"
}
```

- エラー: 401 Unauthorized（X トークン無効）

##### POST /v1/cameras

カメラの初期セットアップで一度だけ呼ばれる。

- Request

```json
{
  "device_name": "My Pi Camera"
}
```

- Response 201

```json
{
  "camera_id": "8a21...",
  "registered_at": "2026-05-13T14:30:00Z"
}
```

##### POST /v1/images

`multipart/form-data` 形式で送信。

- フォームフィールド

| フィールド | 種別 | 必須 | 内容 |
|---|---|---|---|
| `capture_id` | text | ✓ | UUID v4（BLE で受け取った値） |
| `captured_at` | text | ✓ | 撮影日時（ISO 8601 / UTC） |
| `jpeg` | file | ✓ | JPEG 本体 |
| `raw` | file | ✓ | DNG（RAW）本体 |

- Response 201

```json
{
  "image_id": "5c8b...",
  "capture_id": "a1b2...",
  "thumbnail_url": "/v1/images/5c8b.../file?type=thumbnail"
}
```

- エラー: 409 Conflict（capture_id 重複時）

##### GET /v1/images

撮影日時の降順で一覧。

- クエリ
  - `before` (image_id, 任意): この画像より古いものを取得
  - `after` (image_id, 任意): この画像より新しいものを取得
  - `limit` (int, 任意、デフォルト 20、最大 100)

- Response 200

```json
{
  "images": [
    {
      "id": "5c8b...",
      "capture_id": "a1b2...",
      "captured_at": "2026-05-13T14:30:00Z",
      "thumbnail_url": "/v1/images/5c8b.../file?type=thumbnail"
    }
  ],
  "next_cursor": "5c8b...",
  "has_more": true
}
```

##### GET /v1/images/by-capture-id/:capture_id

BLE Capture Done 受信後、アプリが capture_id を元に直接画像を引き当てる。

- Response 200: `GET /v1/images/:id` と同じスキーマ
- エラー: 404（capture_id に対応する画像が未アップロード／存在しない）

##### GET /v1/images/:id

- Response 200

```json
{
  "id": "5c8b...",
  "capture_id": "a1b2...",
  "camera_id": "8a21...",
  "captured_at": "2026-05-13T14:30:00Z",
  "jpeg_url": "/v1/images/5c8b.../file?type=jpeg",
  "raw_url": "/v1/images/5c8b.../file?type=raw",
  "thumbnail_url": "/v1/images/5c8b.../file?type=thumbnail"
}
```

##### GET /v1/images/:id/file

- クエリ
  - `type`: `jpeg` | `raw` | `thumbnail`（必須）
- Response 200: バイナリ本体（`Content-Type: image/jpeg` / `image/x-adobe-dng` / `image/jpeg`）
- 実装は MinIO の presigned URL への 302 リダイレクトでもよい

##### DELETE /v1/images/:id

- Response 204 No Content
- 対応する MinIO 上の JPEG / RAW / サムネもサーバ側で削除する

### 10.11 カメラ初期セットアップフロー

カメラは Rust 製の軽量ファームウェアとして動作し、初回起動時にユーザーが Wi-Fi・X アカウントを設定するための **セットアップモード** を備える。

#### 10.11.1 モード切り替え

- 設定ファイル `/etc/nukon-camera/config.toml` が未作成・不完全な場合は**セットアップモード**で起動
- すべての設定が完了している場合は**通常モード**で起動
- 再セットアップは設定ファイル削除後に再起動で可能（物理スイッチは設けない）

#### 10.11.2 設定ファイル

`/etc/nukon-camera/config.toml` に以下を保持する。

```toml
[server]
url = "http://nukon.local:8080"   # ハードコーディング（変更時は手動編集）

[wifi]
ssid = "..."
# パスワードは別途 wpa_supplicant.conf 等のOS側に保存

[account]
camera_id = "8a21..."             # サーバ登録後に保存
session_token = "eyJ..."          # サーバセッショントークン
x_refresh_token = "..."           # X リフレッシュトークン（暗号化推奨）
```

#### 10.11.3 セットアップフロー

```
[初回起動] → [Wi-Fi設定] → [サーバ接続確認] → [X OAuth ログイン] → [カメラ登録] → [完了]
```

1. **起動 & モード判定**
   - `config.toml` の有無・完全性をチェック
   - 不完全ならセットアップモード

2. **Wi-Fi 設定（カメラの画面とボタンで操作）**
   - 周辺の Wi-Fi SSID 一覧をディスプレイに表示
   - ユーザーがSSIDを選択
   - ソフトキーボードで PW 入力
   - 接続成功で次へ

3. **サーバ接続確認**
   - `config.toml` の `server.url` へ ping
   - 失敗時はエラー表示し再試行を促す

4. **X OAuth ログイン（カメラ内蔵 Web サーバ経由）**
   - カメラ画面に **接続URLとQRコード**（例: `http://<camera-ip>:8080`）を表示
   - ユーザーが別端末（PC / スマホ）のブラウザから接続
   - Web画面の「X でログイン」ボタンを押下
   - PKCE 付き OAuth フローで X の認可ページへ遷移
   - 認可後、カメラの Web サーバにリダイレクトされアクセストークンを受領
   - カメラが `POST /v1/auth/sessions` を呼んでサーバセッションを取得
   - 完了画面を表示
   - セットアップ完了後は Web サーバを停止

5. **カメラ登録**
   - `POST /v1/cameras` を呼び、`camera_id` を受領
   - 受領した `camera_id` / `session_token` / `x_refresh_token` を `config.toml` に保存

6. **完了**
   - 通常モードへ遷移するため再起動（もしくは内部状態を切り替え）

#### 10.11.4 通常モード起動時

- 起動時に `config.toml` を読み込み
- 必要に応じて `x_refresh_token` でアクセストークンを更新し、サーバセッションを再発行
- BLE Peripheral として待ち受け開始
- カメラ画面は撮影モード / 閲覧モードを切り替えるUIに

#### 10.11.5 サブモジュールのペアリング

- 初回セットアップフローには **含めない**。サブモジュールがなくてもメイン本体だけで撮影は完結するため
- ペアリング操作はメインカメラの **設定画面** からいつでも実行可能（手順は [10.14.4](#10144-メインカメラとのペアリング) を参照）

### 10.12 デプロイ・運用

- **開発時**: `docker-compose up` で PostgreSQL と MinIO をローカル起動。サーバ／アプリ／カメラはローカルで個別実行
- **本番時（家サーバ運用）**: Raspberry Pi 上で `docker-compose` で PostgreSQL / MinIO / Go サーバを起動。固定IPまたは mDNS で名前解決
- **カメラ運用**: Raspberry Pi 上で systemd サービスとして Rust バイナリを常駐
- **バックアップ**: MinIO のバケットを外部HDDへ定期コピー（運用は要件外、任意）

### 10.13 画像処理仕様

レタッチ機能で適用するフィルタとパラメータの定義。プレビューは CSS filter で軽量に表示し、最終出力は Tauri Core (Rust) の `image` crate ベースで生成する。

#### 10.13.1 微調整パラメータ

ユーザーがスライダーで操作する基本パラメータ。

| パラメータ | 範囲 | 単位 | 効果 |
|---|---|---|---|
| `exposure` | -100〜+100 | 相対値 | 全体の明るさ（線形ゲイン） |
| `contrast` | -100〜+100 | 相対値 | 明暗差（中央値 0.5 を軸にスケール） |
| `saturation` | -100〜+100 | 相対値 | HSV の S を線形にスケール |
| `temperature` | -100〜+100 | 相対値 | R/B チャンネルをシフト（+で暖色） |
| `tint` | -100〜+100 | 相対値 | G チャンネルをシフト（+で緑、-でマゼンタ） |

数値は内部で `[-1.0, +1.0]` に正規化して適用する。

#### 10.13.2 プリセットフィルタ

各プリセットは「**ベースの数値パラメータ** + **追加エフェクト**」で定義する。

##### トイカメラ風（`toy_camera`）

```toml
saturation = +35
contrast   = +25
temperature = +10

[effects]
vignette = { strength = 0.5, radius = 0.7 }  # 周辺減光
```

##### シネマ・モノクロ調（`cinema_mono`）

```toml
saturation = -100   # 完全モノクロ
contrast   = +30

[effects]
tone_curve = "high_contrast_filmic"  # シャドウを締めハイライトを伸ばす
```

##### フィルム調（`film`）

```toml
saturation = -10
contrast   = +20
temperature = +5

[effects]
color_shift = { channel = "shadow_blue", amount = 0.15 }  # シャドウに青み
grain = { amount = 0.08, monochrome = false }              # 粒子ノイズ
```

#### 10.13.3 エフェクトのアルゴリズム

##### vignette（周辺減光）

- 画像中心からの距離 `d` に応じて輝度を減衰
- `strength`: 端の暗さ（0.0〜1.0）
- `radius`: 減光が始まる中心からの距離比（0.0〜1.0）
- 実装: 各ピクセルで `factor = 1 - clamp((d - radius) / (1 - radius), 0, 1) * strength` を乗算

##### tone_curve

- ハイライト・シャドウのカーブを変える非線形マッピング
- プリセット名で複数のLUT風カーブを定義（例: `high_contrast_filmic`, `flat_low_contrast`）
- 実装: 1次元 LUT を Rust 側で事前定義し、各チャンネルに適用

##### color_shift

- 指定領域（`shadow` / `midtone` / `highlight`）のRGBバランスをシフト
- `channel = "shadow_blue"` の場合、輝度の低いピクセルに青を足す
- 実装: 各ピクセルの輝度から領域マスクを計算し、対象チャンネルへ係数を加算

##### grain（粒子）

- ピクセルごとに微小なノイズを加算
- `amount`: ノイズの強度（0.0〜0.3）
- `monochrome`: true なら全チャンネル同値、false ならチャンネル別ランダム
- 実装: シード固定でも可（同じ画像に再適用しても同じ結果）

#### 10.13.4 適用順序

```
[元画像]
   │
   ▼ exposure → contrast → tone_curve → saturation → temperature → tint
   │
   ▼ color_shift → vignette → grain
   │
[出力]
```

- 微調整パラメータ → エフェクトの順
- エフェクトは tone系 → 色系 → 空間系（vignette） → 粒子の順

#### 10.13.5 プレビューと最終出力の使い分け

| 段階 | 処理場所 | 内容 |
|---|---|---|
| スライダー操作中のライブプレビュー | Web フロント（CSS filter） | `brightness()` `contrast()` `saturate()` `hue-rotate()` `sepia()` などで近似。vignette は `box-shadow inset`、grain は SVG noise で代替 |
| 「適用」ボタン押下時の最終出力 | Tauri Core（Rust） | `image` crate で元解像度に対し正確な処理を実行 |
| サーバアップロード前のサムネ生成 | Tauri Core（Rust） | 最終出力からリサイズ（例: 長辺 512px） |

CSS filter は近似であり最終結果と完全には一致しない旨を UI 上で示す（操作感は十分得られる前提）。

#### 10.13.6 出力仕様

- 形式: **JPEG（quality 90）**
- メタデータ: 元画像の `capture_id` を `parent_image_id` として保持し、サーバへは `kind="retouched"` でアップロード
- レタッチ結果は **常に元画像とは別ファイル**（非破壊編集）

### 10.14 サブモジュール（操作機）仕様

メイン本体から離れた位置で撮影操作を行うための小型リモコン。BLE Central としてメイン本体に接続する。

#### 10.14.1 役割

- 2段シャッターボタンを物理操作してメインカメラへ Half / Full の指示を送る
- 十字キーでメインカメラのナビゲーション操作を送る
- 自身のバッテリー残量を定期的にメインカメラへ通知する

#### 10.14.2 ハードウェア構成

| 項目 | 採用品／仕様 |
|---|---|
| MCU | ESP32 系（Wi-Fi/BLE 内蔵）または Raspberry Pi Pico W |
| シャッターボタン | 物理2段スイッチ（半押し・全押しを別接点で検出） |
| 十字キー | 4方向 + 中央押し込みのタクトスイッチ群 |
| 電源スイッチ | スライドまたはトグル式 |
| バッテリー | リチウムイオン（CR2032 級では駆動時間が足りないため二次電池を想定） |
| 充電端子 | USB-C |
| 状態表示 | LED（接続状態・電池残量警告） |
| 重量目標 | 100g 以下 |

#### 10.14.3 ソフトウェア構成

- 言語: **Rust（Embedded Rust, `esp-hal` 等）** を推奨。代替として ESP-IDF（C）も可
- BLE 役割: **Central**。メインカメラ（Peripheral）の Camera Control Service へ接続
- 主要処理:
  - 起動 → メインカメラへの自動再接続
  - ボタン押下を割り込みで検出し、対応する BLE Write を発行
  - 60秒ごとに自身のバッテリー残量を Submodule Battery へ Write
  - 一定時間無操作でスリープ／低消費モードへ

#### 10.14.4 メインカメラとのペアリング

- 初回ペアリングはメインカメラの **設定画面** から「サブモジュールをペアリング」を選び実行する
- サブモジュールは起動後にアドバタイズ／スキャン中であれば、メインカメラ画面で対象が表示される
- 認可するとペアリング情報を双方の不揮発領域に保存し、以降は自動接続
- 解除はメインカメラの設定画面から実行

#### 10.14.5 ボタン → BLE のマッピング

| 物理操作 | BLE 動作 |
|---|---|
| シャッター 半押し | Shutter Half ← 0x01 |
| シャッター 全押し | Shutter Full ← 0x01 |
| シャッター 解放 | Shutter Half ← 0x00 |
| 十字 上 | Navigation Key ← 0x01 |
| 十字 下 | Navigation Key ← 0x02 |
| 十字 左 | Navigation Key ← 0x03 |
| 十字 右 | Navigation Key ← 0x04 |
| 十字 中央押し込み | Navigation Key ← 0x05 |
