# スマートグラス AIセカンドブレイン 要求仕様書

## 1. 基本情報

| 項目 | 内容 |
| --- | --- |
| 文書名 | スマートグラス AIセカンドブレイン 要求仕様書 |
| 対象アプリ | 人物中心のAIセカンドブレイン |
| 対象デバイス | Ray-Ban Meta glasses + Android端末 |
| 開発者 | 小松琢磨、中村壮馬 |
| 初期開発方針 | Android/Kotlinを主実装とし、ボタン操作でその時点のカメラ画像と音声を自作AIへ送り、AI記憶として保存する機能を優先する |

## 2. 目的

本アプリは、Ray-Ban Metaのカメラと音声入力をAndroidアプリから扱い、ユーザーがボタンを押したタイミングの画像内容と音声を自作AIへ送信して記憶させることで、ユーザーの対人記憶と視界記憶を補助する。

初期MVPでは、常時録画や高精度なAR位置合わせは目標としない。ユーザーがAndroidアプリ上の保存ボタン、通知アクション、または利用可能なグラス側操作を押したときだけ、現在の画像と直近の音声バッファをAI記憶として保存する。人物登録については、AIが画像と音声から人物候補、物体、場所、会話内容を抽出し、必要に応じてユーザー確認を挟みながら記憶を作成する。

## 3. スコープ

### 3.1 MVPで実装する範囲

- カメラ画像から人物を検出する。
- ユーザーがボタンを押したタイミングで現在の画像を保存対象にできる。
- ユーザーがボタンを押したタイミングで直近の音声またはその時点からの音声を保存対象にできる。
- 保存対象の画像と音声を自作AIサーバーへ送信し、内容要約とタグを生成できる。
- 自作AIが生成した記憶を後から検索できる形で保存できる。
- 検出した人物を一時的な人物候補として扱う。
- 会話音声から名前、所属、関係性、話題、役割の候補を抽出できる。
- 検出人物と会話コンテキストを紐付け、人物候補を自動生成できる。
- ユーザーが人物候補を後から確認・修正・確定できる。
- 登録済み人物の顔画像または人物特徴量を保存できる。
- 登録済み人物とカメラに映った人物を照合できる。
- 登録済み人物に紐付く情報をスマホ画面、通知、音声で提示できる。
- ユーザーが人物情報を音声または簡易UIで検索できる。
- 人物情報、顔画像、接触履歴、メモを削除できる。

### 3.2 MVPで実装しない範囲

- ユーザー確認なしの完全自動確定登録。
- 厳密な視線追跡。
- 現実空間上の人物頭上へのミリ単位のAR固定。
- 常時録画による全行動ログ保存。
- ユーザー操作なしでの常時AI記憶化。
- 多人数環境での完全な個人識別。
- ゴースト・トレーナー、遠隔作業支援の本実装。

### UC-000 ボタン操作で今見ている内容をAIに記憶させる

| 項目 | 内容 |
| --- | --- |
| アクター | 装着者 |
| 前提 | Androidアプリが起動しており、カメラ・音声取得とAIサーバー送信の権限が許可されている |
| トリガー | 装着者がAndroidアプリの保存ボタン、通知アクション、または利用可能なグラス側操作を押す |
| 基本フロー | 1. アプリが現在のカメラ画像を取得する。2. アプリが直近の音声バッファまたはボタン押下後の音声を取得する。3. アプリが画像、音声、時刻、任意の位置情報を自作AIサーバーへ送信する。4. AIが画像内容、会話内容、人物候補、物体、場所、文字、要約、タグを生成する。5. アプリが生成結果をAI記憶として保存する。6. アプリが保存完了をスマホ画面、通知、音声のいずれかで返す。 |

## 6A. 人物登録・候補生成詳細

### 6A.1 人物状態

人物情報は、誤登録と誤表示を避けるため、以下の状態で管理する。

| 状態 | 説明 | 情報提示 | 保存 |
| --- | --- | --- | --- |
| Detected | カメラに人物または顔が映っただけの状態 | 原則表示しない | 一時データのみ |
| Candidate | 顔情報、会話、文字認識などから人物候補が作成された状態 | 候補としてのみ表示可 | 未確定候補として保存 |
| Confirmed | ユーザー確認済みで、登録済み人物として扱える状態 | 通常ラベル表示可 | 人物DBに保存 |
| Discarded | ユーザーが不要と判断した候補 | 表示しない | 削除または破棄ログのみ |
| Merged | 既存人物に統合された候補 | 統合先人物として表示 | 統合履歴を保存 |

状態遷移は以下を基本とする。

```text
Detected -> Candidate -> Confirmed
                     -> Discarded
                     -> Merged
```

### 6A.2 会話セッション開始条件

アプリは常時すべての音声を人物候補化対象にしない。MVPでは、人物検出と音声活動が重なった場合に会話セッションを開始する。

| ID | 条件 |
| --- | --- |
| CS-START-001 | 人物または顔が一定時間以上カメラに映っている |
| CS-START-002 | 音声入力に一定以上の発話らしき信号がある |
| CS-START-003 | 人物検出と音声活動が同じ時間帯に発生している |
| CS-START-004 | ユーザーが明示的に停止していない |

MVPでは、具体的なしきい値は実機検証で調整する。初期値の目安は、人物検出が2秒以上継続し、音声活動が1秒以上検出された場合とする。

### 6A.3 会話セッション終了条件

| ID | 条件 |
| --- | --- |
| CS-END-001 | 検出人物が画面外に出た |
| CS-END-002 | 一定時間以上、発話が検出されない |
| CS-END-003 | セッションの最大継続時間に達した |
| CS-END-004 | ユーザーが明示的に停止した |
| CS-END-005 | カメラ、音声、権限、バッテリーなどの制約で継続できない |

MVPでは、無音終了の目安を10秒から20秒、最大継続時間の目安を5分とする。実機の発熱、バッテリー、音声認識コストに応じて調整する。

### 6A.4 人物候補生成条件

会話セッション終了時、または一定間隔で、以下の条件を満たす場合に人物候補を生成する。

| ID | 条件 | 説明 |
| --- | --- | --- |
| PC-CREATE-001 | 名前候補が抽出された | 「〇〇です」「〇〇さん」「株式会社〇〇の〇〇」など |
| PC-CREATE-002 | 所属または役割候補が抽出された | 会社名、学校名、役職、担当、関係性など |
| PC-CREATE-003 | 同一人物が一定時間以上映っていた | 一瞬映っただけの人物を除外する |
| PC-CREATE-004 | 会話が一定時間以上続いた | 短い環境音や通りすがりを除外する |
| PC-CREATE-005 | 名刺、名札、スライドなどから人物関連テキストが抽出された | OCR結果を人物候補に利用する |

人物候補は、上記のうち少なくとも1つ以上の強い手がかりがある場合に作成する。強い手がかりがない場合は、検出イベントとして一時保存するだけに留める。

### 6A.5 人物候補を生成しない条件

| ID | 条件 | 説明 |
| --- | --- | --- |
| PC-SKIP-001 | 人物が一瞬しか映っていない | 通行人などを候補化しない |
| PC-SKIP-002 | 音声が環境音、動画、店内放送などである可能性が高い | 会話相手と関係しない音声を除外する |
| PC-SKIP-003 | 複数人が映っており、誰の情報か判定できない | 不確実な紐付けを避ける |
| PC-SKIP-004 | 顔画像の品質が低すぎる | 暗い、横顔、ブレ、遮蔽など |
| PC-SKIP-005 | ユーザーが候補生成を一時停止している | プライバシーや場面への配慮 |

### 6A.6 未確定候補の確認タイミング

会話中に確認を強く割り込ませると自然な対話を妨げるため、確認は以下の優先順位で行う。

| 優先度 | タイミング | 説明 |
| --- | --- | --- |
| 高 | 会話終了後 | 人物が画面外に出た、または無音が続いた後に短く提示する |
| 中 | 後で管理画面 | PCまたはスマホの候補一覧でまとめて確認する |
| 低 | 会話中 | 信頼度が非常に高く、表示しても邪魔にならない場合のみ小さく提示する |

会話中に出す場合も、入力を強制しない。スマホ通知または音声では「候補を作成しました」程度の短い通知に留める。

### 6A.7 未確定候補の表示ルール

未確定候補は、確定人物と同じ見た目で表示しない。低信頼度情報を断定しないため、表示文言を明確に分ける。

| 状態 | 表示例 | 備考 |
| --- | --- | --- |
| Candidate | 候補: 田中さん? | 推定名を疑問形で表示 |
| Candidate | 田中さんかも / 株式会社X | 所属も推定の場合は短く表示 |
| Confirmed | 田中さん | 確定済み人物のみ断定表示 |
| Confirmed | 田中さん / 前回: 5/22 / メモ: 展示会で会話 | 追加情報は短く表示 |
| Unknown | 不明人物 | 候補がない場合 |

### 6A.8 自動候補生成の安全ルール

| ID | ルール |
| --- | --- |
| SAFE-REG-001 | ユーザー確認前の人物候補を確定人物として扱わない |
| SAFE-REG-002 | 低信頼度の名前、所属、関係性を断定表示しない |
| SAFE-REG-003 | 複数人環境で発話者と人物の対応が曖昧な場合、候補を未確定のまま保持する |
| SAFE-REG-004 | ユーザーは候補生成、音声取得、情報提示を一時停止できる |
| SAFE-REG-005 | ユーザーは未確定候補を一括削除できる |
| SAFE-REG-006 | 顔画像、特徴量、会話要約、文字起こしは削除対象に含める |

## 13. 開発順序

1. Android/Kotlinプロジェクトを作成する。
2. スマホカメラとスマホマイクで保存ボタン押下時のMemoryEvent作成を実装する。
3. MemoryEventを自作AIサーバーへ送信するAPIクライアントを実装する。
4. AI解析結果を保存し、一覧・詳細・削除できる管理画面を実装する。
5. 自然言語検索またはテキスト検索で保存済み記憶を探せるようにする。
6. Meta Wearables Device Access Toolkitを導入し、Ray-Ban Metaとの接続を確認する。
7. Ray-Ban Metaのカメラ・音声をMemoryEvent入力へ差し替える。
8. 会話音声の文字起こしと名前・所属・話題抽出を実装する。
9. 検出人物と会話コンテキストから人物候補を作成する。
10. 人物候補の確認・修正・確定フローを実装する。
11. 登録済み人物との照合処理を実装する。
12. 通知アクションまたは利用可能なグラス側操作から保存できるようにする。

## 14. 技術選定・実装方針

### 14.1 基本方針

MVPはAndroid/Kotlinで実装する。Ray-Ban Metaにはディスプレイがないため、スマートグラス上の表示やAR表現ではなく、Androidアプリからカメラ・音声を取得して自作AIへ送信する処理を主軸にする。

初期段階ではスマホカメラ・スマホマイクのモック実装から始め、Meta Wearables Device Access Toolkitの接続確認後にRay-Ban Meta入力へ差し替える。

参考:

- [Meta Wearables Device Access Toolkit for Android](https://github.com/facebook/meta-wearables-dat-android)
- [Introducing the Meta Wearables Device Access Toolkit](https://developers.meta.com/blog/introducing-meta-wearables-device-access-toolkit/)

### 14.2 使用言語・技術

| 領域 | 採用方針 | 備考 |
| --- | --- | --- |
| メインアプリ | Android / Kotlin | MVPの主実装 |
| UI | Jetpack Compose | 保存ボタン、記憶一覧、検索、設定 |
| デバイスSDK | Meta Wearables Device Access Toolkit | Ray-Ban Metaのカメラ・音声取得の確認対象 |
| カメラ入力 | DAT Camera / CameraX | 初期はCameraXモック、後続でRay-Ban Meta入力 |
| 音声入力 | DAT Audio / Android AudioRecord | 初期はスマホマイク、後続でRay-Ban Meta入力 |
| AI処理 | 自作AIサーバー/API連携 | 画像説明、文字起こし、要約、記憶保存、検索 |
| 顔・人物検出 | 自作AIサーバー、MediaPipe、ML Kit等 | 実機性能と精度で選定 |
| データ保存 | Room / SQLite + ローカルファイル | MemoryEvent、人物DB、候補DB、接触履歴 |
| 管理画面 | Androidアプリ内画面 | 記憶一覧、人物候補確認、削除 |

### 14.3 Android/Kotlin採用理由

| 理由 | 内容 |
| --- | --- |
| デバイス連携 | Ray-Ban MetaはAndroid/iOSアプリからの連携が前提になる |
| 表示前提 | 対象グラスにディスプレイがないため、スマホUIと音声応答が主UIになる |
| カメラ・音声 | CameraX、AudioRecord、権限、通知、バックグラウンド処理を直接扱いやすい |
| SDK整合 | Meta Wearables Device Access Toolkit for Androidをそのまま導入できる |
| 検証速度 | 実機がなくてもスマホカメラ・スマホマイクでAI記憶化フローを先に作れる |

### 14.6 音声バッファ方針

ボタンを押した瞬間だけ録音を開始すると、押す直前の会話を取り逃がす可能性がある。MVPでは、ユーザーが明示的に有効化したときだけ短時間のリングバッファを保持する方式を検討する。

| 方針 | 内容 |
| --- | --- |
| 初期 | ボタン押下後に録音を開始し、数秒から数十秒を保存する |
| 改善 | ユーザー許可時のみ、直近10秒から30秒の音声リングバッファを端末内に一時保持する |
| 保存 | ボタン押下時に必要部分だけMemoryEventへ固定する |
| 削除 | 保存しなかったリングバッファは短時間で破棄する |
